## 重要な observations

### 1. Tool errors の差が最強の signal

各 trial の tool-errors 数:
- OFF: [1, 1, 3, 1, 1, 2, 1, 1, 1, 2] (sum=14, mean=1.4)
- ON: [3, 4, 4, 2, 2, 4, 3, 3, 2, 5] (sum=32, mean=3.2)

**全 10 trial で ON ≥ OFF**。これは variance ではなく **agent prompt が systematic に
出力品質を劣化させている** 証拠。Wilcoxon signed-rank も significant (実数値計算は省略)。

### 2. ON trial 10 で #45 limit が初発火

```
trial 10: status=limit-exhausted limit=max-consecutive-failed-patches ok=0 fail=3 tool_err=5
```

**3 consecutive failed patches** で #45 (max-consecutive-failed-patches=3) が triggered。
これは #45 が想定した「patch 戦略 die-spiral」が実在することの直接的証拠。
**#38 が #45 を発火させる pathology を作っている** 関係も観察。

### 3. #38 directive 不遵守は ON でも継続

全 ON trial で agent は依然 evolved-failure pattern (cache-put → verify-fail with cache-get
undefined → cache-get → pass) を辿った。**#38 が "implement ALL in SAME patch" と指示しても
Qwen3.6 は遵守せず**、代わりに以下の副作用が出る:
- 出力 patch の構文/JSON 品質が低下 (tool-errors +128%)
- 同 evolved pattern でも余分な turn / tool-call を消費

### 4. 仮説 (refinement)

Qwen3.6 が "Patching guidance" を読んだ時:
1. "全 symbol を 1 patch でまとめる" は理解する
2. 但し agent workflow は既に決まっていて (read → patch → verify)、prompt 1 段落で変えられない
3. 結果として agent は同じ incremental approach を取るが、prompt が複雑化したため
   各 turn の output 品質が劣化 (JSON parse error / token-match miss が増える)
4. → tool-errors 増加 → 余分な turn → wall-clock 増加 → pass-rate 低下

つまり **prompt の "noise" 効果**であって "behavior 修正" 効果ではない。

## 推奨処置: #38 段落を削除

N=10 evidence:
- Pass rate: -20pt (CI overlap だが consistent direction)
- Tool errors: +128% (highly significant, all-trial monotone)
- Wall-clock: +12% (consistent)
- Directive 自体は遵守されていない

**netnegative 確定**。当面の処置として **#38 patching-guidance 段落を削除** を推奨。
削除すれば:
- 即時 effect: 全 develop run で tool-errors / wall-clock 低下
- prompt 簡素化 → token budget 節約 → context-tokens の余裕も増える
- 副作用なし (#45 limit は維持されるので die-spiral 対策は別 layer で残る)

代替案 (削除しない場合):
- 段落の wording を弱める ("implement ALL" → "consider whether to add related symbols
  together" 等)。但し効果未検証
- N=20+ で再評価 (但し既に N=10 で tool-errors の差は decisive なので追加データの
  情報量は限定的)
