## Observations

### Conditions

- **`file-only`** — 0 / 4. Lacks lisp-aware patching; the agent can only
  read a file and rewrite it whole via `fs-write-file`. Across all four
  tasks the LLM produced 2-3 attempted full-file rewrites that never
  satisfied the test. Worth noting: the LLM occasionally proposed a
  correct body but mis-spelled the package or mis-aligned the
  `(in-package ...)` form when re-emitting the whole file. **The
  baseline performs as expected: it surfaces the fragility of
  whole-file rewriting on a real LLM.**
- **`generic-mcp`** — 3 / 4. Structured `lisp-patch-form` /
  `lisp-edit-form` calls fix the typo / wrong-operator / missing-greet
  cases on the first or second attempt.
- **`runtime-native`** — 3 / 4. Same outcomes as generic-mcp on this
  small sample. Phase 3.6 schema hints already keep the patch shape
  correct; the extra `repl-eval` / `code-describe` tools didn't shave
  turns on these particular bugs (the failure path is "read source →
  patch", not "probe runtime"). On 001-wrong-operator the runtime-native
  run was slightly faster (7 turns vs 8), within noise.

### Tasks

- **000-smoke / 001-wrong-operator** — fixed in 1 patch by both
  cl-mcp conditions. Both bugs are local body changes. Schema hints
  carry the LLM directly.
- **002-typo-defun** — both cl-mcp conditions land the fix but burn
  ~13 turns and 7 patch attempts. The LLM's first reflex is to add a
  new `defun greet` instead of renaming `gret` to `greet`, which
  produces multiple definitions and sometimes loops. The fix path
  itself works; the LLM heuristic is suboptimal.
- **003-missing-import** — neither cl-mcp condition fixes it within
  20 turns. The bug is in `tests/main-test.lisp`'s `:import-from`
  clause (typo `add-tow` instead of `add-one`); the LLM repeatedly
  reads `src/main.lisp` looking for the function, attempts patches
  that don't land on the import line, and exhausts max-turns.
  Hypothesis: the initial verify reports `:load-failed` but the
  detail block (`Load failure detail: ...`) needs to point the agent
  to the `:import-from` line more explicitly, e.g. by pulling the
  exact "no symbol named X in PKG" message out of the load output
  and surfacing it in the system prompt.

### Methodology notes

- Per-task `pool-kill-worker reset=t` was added in Phase 5.0 after
  the first run produced an anomalous turn-0 `:passed` for
  `002-typo-defun × runtime-native`: the previous condition had
  patched the same package, leaving the corrected `greet` function
  bound in the same worker process. Without this isolation, a
  benchmark across conditions risks measuring "did the previous run
  leak state?" instead of the agent's actual capability.
- `:clean-verify-p nil` is passed to `run-agent` for benchmark runs.
  The sandbox is registered via `asdf:initialize-source-registry`
  with `:ignore-inherited-configuration`; pool-kill mid-run would
  drop that registration and revert to the pristine fixture in the
  repo. Production `fix` runs (CLI) keep clean-verify enabled.

## Limitations

- **N=4 tasks is small.** PRD §17 MVP target is 10+ tasks. Phase 5.1
  must add 6 more.
- **Single model.** All numbers are llama-3.3-70b-versatile. Other
  models would show how condition advantages depend on model
  capability.
- **Single trial per (task × condition).** Variance is unmeasured.
  Phase 5.1 should run each cell 3+ times and report mean ± std.
- **No per-task token budget.** Two tasks hit max-turns at 20.
- **Score derivation from transcripts is heuristic** in this report
  because the suite hit the 600s tool-call timeout on the first
  pass; the data here was reconstructed from JSONL events plus a
  single re-run for the missing 12th datapoint.
