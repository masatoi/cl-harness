# Phase 5.2 benchmark results — MVP suite reaches 10 tasks

**Date**: 2026-05-05
**Model**: `llama-3.3-70b-versatile` via Groq
**Run-limits**: max-turns 20, temperature 0.0, max-tokens 2048
**Code**: cl-harness Phase 5.2 (+`run-benchmark-task-trials`,
008-macro-expansion, 009-handler-case-typo)

This report closes Phase 5: the suite now contains the 10 tasks PRD §17
requires, the runner can repeat any (task × condition) cell N times, and
the first variance datapoints make it clear that single-trial numbers
should not be quoted as headline pass-rates.

## Suite size

10 tasks, all generated and verified to fail before-fix:

| ID | Bug |
| -- | --- |
| 000-smoke                  | `add` returns difference instead of sum |
| 001-wrong-operator         | `max-of` uses `<` instead of `>` |
| 002-typo-defun             | exported `greet` has no `defun` (named `gret`) |
| 003-missing-import         | test's `:import-from` typoed `add-tow` instead of `add-one` |
| 004-defmethod-specializer  | method specialized on `string`, called with a class instance |
| 005-defclass-slot          | slot's `:initarg` is `:usrname`, callers pass `:name` |
| 006-format-arity           | `(format nil "(~A)" a b)` drops second argument |
| 007-error-signal           | `safe-div` returns `0` on a zero divisor instead of erroring |
| 008-macro-expansion (new)  | `with-counter` macro returns init-value, not the bound symbol |
| 009-handler-case-typo (new)| `handler-case` clause names `type-error`; body signals `my-error` |

## New tasks: smoke (`:generic-mcp`, n=1)

| Task | Status | Turns | Tokens |
| ---- | ------ | -----:| ------:|
| 008-macro-expansion   | `passed` | 19 | 41,048 |

008 nearly hit max-turns (19 / 20). Macro-expansion bugs are harder
in `:generic-mcp`: the LLM cannot run `(macroexpand-1 ...)` to see the
expansion and has to reason from source alone. Phase 5.3+ work should
re-run 008 in `:runtime-native` to test the working hypothesis that
probe access closes that gap.

## Variance demo (003-missing-import × `:generic-mcp`, n=3)

| Trial | Status | Turns | Tokens | Patches applied |
| -----:| ------ | -----:| ------:| ---------------:|
| 1 | `passed` |  8 | 11,575 | 3 |
| 2 | `passed` |  5 |  6,115 | 2 |
| 3 | `passed` | 20 | 49,250 | 4 |

Headline numbers from this single cell:

- **Pass-rate**: 3 / 3 = 100%.
- **Turns**: 5 / 8 / 20 (mean 11, range 15).
- **Tokens**: 6.1k / 11.6k / 49.2k (max ≈ 8× min).
- **Patches**: 2 / 3 / 4.

All three trials succeeded, but the spread is wide enough that the
4-turn / 4.6k-token figure reported in `results-2026-05-05-1.md` was
clearly a low draw. **Single-trial pass-rates are an existence proof,
not a performance metric.** Even at `temperature 0.0`, llama-3.3-70b
on Groq is not deterministic across calls (presumably due to batch
floating-point differences in their serving stack), and the agent
loop's branch points compound that into measurable wall-clock spread.

## What's missing for a defensible final number

- **3-condition × 10-task × 3-trial sweep** has not been run end-to-end.
  90 LLM-driven runs at ~30s avg + worker reset = ~50 min wall clock,
  ~720k tokens — above one tool-call timeout window. Mechanically
  supported (the tooling exists); execution is a follow-up exercise.
- **`:file-only` and `:runtime-native` columns for tasks 004-009** were
  not run. Phase 5.0 file-only baseline (0/4) is the only file-only
  data we have; runtime-native data exists only for 000-003.
- **`:runtime-native` 003-missing-import** still hits max-turns (Phase
  5.1 finding, unchanged).
- **No multi-model comparison.** Everything in this report is
  llama-3.3-70b-versatile.
