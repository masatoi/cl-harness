## なぜ wall-clock / pass-rate が改善したか

#45 が triggered していないので、improvement は他要因:

1. **Variance (planner output の運)**: planner が同 fixture でも run 毎に異なる plan を
   生成 (本 sweep でも 102 trial1: 5 step / trial2: 5 step / trial3: 4 step と plan 構造異なる)
2. **server / network 条件**: Qwen endpoint の load や latency が時間帯で変動
3. **subtle cumulative effects from prior commits**: 直近の spec-generator robustness
   (fc9c7c0) と CLI max-tokens fix (93e6648) が今 sweep で有利に作用した可能性

**正直な評価**: 本 9 cell sample では「#45 の effect」と「variance」を分離する根拠なし。

## 102-counter-class の 3/3 PASSED は本当に意味があるか

N=3 ですべて PASSED は pre-sweep の 1/3 (PASSED 821s, LIMIT 1968s, GIVE-UP 350s) と
比較して劇的だが、本 commit (fd02eaa) は #45 以外にも変更を含むため (spec-generator /
CLI max-tokens fix)、102 の改善は #45 単独の effect ではない可能性が高い。

ただし pre-sweep で 102 が苦戦した理由は planner test stub の MOP 関数 (`class-slots`)
含有や agent patch の連続 ok=False だった。本 sweep でこれが解消されたのは:
- planner output の品質変動 (luck)、または
- spec-generator robustness が planner phase にも波及効果を持った (unlikely)

## 実 effect 測定のために必要なこと

#45 の真の effect を測るには:

1. **N≥10 sweep**: variance を抑え、たまの pathological run で #45 が fire するかを観察
2. **fixed-plan paired bench (backlog #46)**: 同 plan で #45-on/off 比較
3. **failure-mode injection**: 故意に malformed patch を出す mock agent で #45 path を強制 fire

今回データはあくまで「#45 を入れた後の sweep の数値」で、causation は弱い。但し:
- code path として #45 は正しく実装 (test 464 passed)
- defensive backstop として存在することのコストは小さい (zero impact when not triggered)
- 将来 pathological run が起きた時に救済する保険として価値あり
