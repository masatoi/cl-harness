# cl-harness scaffold コマンド設計

**Date:** 2026-05-22
**Author:** Satoshi Imai
**Status:** Draft for implementation
**Related:** (planned) `2026-05-22-planner-symbols-to-export-design.md`

## 1. 動機

現状 `cl-harness develop` は `--system` / `--test-system` / `--test-file` を必須引数で受けており、それらが指す ASDF プロジェクト一式 — `<system>.asd` + `src/main.lisp` の defpackage + `tests/main-test.lisp` の rove defpackage — が予め用意されている前提で動く。

`develop-benchmarks/100-greet/fixture/` 等は手書きでこの骨格を整えてあるが、新規プロジェクトに対して develop を回したいユーザはまずこれを自分で書く必要がある。本 spec は **その骨格生成だけを担う独立コマンド `cl-harness scaffold`** を導入する。

### 1.1 設計判断: なぜ独立コマンドか

`develop` 内に "auto-scaffold" として埋め込む案も検討したが、

- `--system` / `--test-system` / `--test-file` 引数は「scaffold 済み」を前提とする契約であり、内蔵すると引数の required 集合が状況依存になり API が濁る。
- scaffold は one-shot setup、develop は反復操作で、責務の粒度が異なる。
- Common Lisp 圏には `quickproject:make-project` 等の先例があり、Lisper の既存メンタルモデルと整合する。
- scaffold を LLM ゼロにすることで CI で LLM 鍵なしの健全性確認に使える。

ゆえに **scaffold は独立コマンド**、`develop` の契約は不変とする。

## 2. コマンドインタフェース

```
cl-harness scaffold --project-root DIR --system NAME
                    [--test-system NAME] [--test-file PATH] [--force]
```

### 2.1 必須引数

| 引数 | 意味 |
|---|---|
| `--project-root DIR` | 出力先ディレクトリ。存在しなければ `uiop:ensure-all-directories-exist` で作る。同名のファイル(非ディレクトリ)が存在した場合は UIOP が raise する error を CLI が exit 2 (unknown failure) に変換する。 |
| `--system NAME` | ASDF system 名。同時に main package の nickname にもなる。正規表現 `^[a-z][a-z0-9-]*$` を強制(満たさなければ exit 2)。 |

### 2.2 任意引数(`--system` から自動導出)

| 引数 | 既定値 |
|---|---|
| `--test-system NAME` | `<system>/tests` |
| `--test-file PATH` | `<project-root>/tests/main-test.lisp` |
| `--force` | NIL — 部分存在の refuse を上書きする。backup は取らない(ユーザ責任)。 |

### 2.3 プログラム API

```lisp
(cl-harness:scaffold :project-root "/tmp/demo/"
                     :system "demo"
                     :test-system "demo/tests"   ; optional
                     :test-file nil               ; optional, derived
                     :force nil)                  ; optional
;; → SCAFFOLD-RESULT instance
```

戻り値 `scaffold-result` クラスのスロット:

- `status` — `:written` / `:already-present` / `:refused` の keyword
- `paths-written` — 実際に書いたファイルパスのリスト(`:already-present` / `:refused` 時は NIL)
- `conflicts` — `:refused` 時に「既存だったファイル」のリスト、それ以外は NIL

## 3. 生成ファイル(4 つ)

### 3.1 `<root>/<system>.asd`

`develop-benchmarks/100-greet/fixture/greet.asd` を `<system>` でパラメタライズしたもの。`:class :package-inferred-system` + test-system の `:perform (test-op ...)` は rove discovery (prefix `<system>/tests/`) を踏襲。

```lisp
(asdf:defsystem "<system>"
  :class :package-inferred-system
  :description "TODO: short description"
  :license "MIT"
  :version "0.0.1"
  :depends-on ("<system>/src/main")
  :in-order-to ((test-op (test-op "<test-system>"))))

(asdf:defsystem "<test-system>"
  :class :package-inferred-system
  :depends-on ("rove"
               "<system>"
               "<test-system>/main-test")
  :perform (test-op (o c)
                    (declare (ignore o))
                    (let ((test-packages
                           (remove-if-not
                            (lambda (dep)
                              (and (stringp dep)
                                   (uiop:string-prefix-p "<test-system>/" dep)))
                            (asdf:system-depends-on c))))
                      (uiop:symbol-call :rove :run test-packages))))
```

### 3.2 `<root>/src/main.lisp`

```lisp
;;;; <relative-path>
;;;;
;;;; Generated by cl-harness scaffold.

(defpackage #:<system>/src/main
  (:nicknames #:<system>)
  (:use #:cl)
  (:export))

(in-package #:<system>/src/main)
```

`(:export)` は意図的に空。シンボルの export は planner / executor 側の責務(別 spec `2026-05-22-planner-symbols-to-export-design.md` で扱う)。

### 3.3 `<root>/tests/main-test.lisp`

```lisp
;;;; <relative-path>
;;;;
;;;; Generated by cl-harness scaffold.

(defpackage #:<test-system>/main-test
  (:use #:cl #:rove))

(in-package #:<test-system>/main-test)
```

`<test-system>/main-test` パッケージ名は `<system>.asd` の test-system の `:depends-on` リストと一致させる(`<test-system>/main-test` 名が package-inferred-system による file→package マッピングで `tests/main-test.lisp` を指す)。

### 3.4 `<root>/.gitignore`

3 行のみ:

```
*.fasl
*.fasl-tmp
.cache/
```

### 3.5 意図的に出さないもの

- `README.md` — ユーザ独自の説明があるべき
- `LICENSE` — ライセンス選択はユーザ責任
- `.mallet/` 等の lint config — プロジェクト全体規約に依存
- `CI/CD` 設定 — 同上

## 4. 振る舞いマトリクス

detection 対象は §3.1–3.3 の 3 ファイルのみ(`.gitignore` は detection に含めない)。

| 既存状態 | 既定動作 | exit code | `--force` 動作 |
|---|---|---|---|
| 3 つすべて不在 | 4 ファイル書く、`:written` | 0 | 同左 |
| 3 つすべて存在 | no-op、`:already-present` | 0 | 4 ファイル上書き、`:written` |
| 部分存在(1 or 2 個) | refuse、stderr に存在/不在列挙、`:refused` | 1 | 4 ファイル上書き、`:written` |
| `--system` 不正 | `scaffold-bad-system-name` raise → CLI が exit 2 に変換 | 2 | 同左 |
| `--project-root` 不在 | `uiop:ensure-all-directories-exist` で作る | (以降のロジックに従う) | 同左 |

`--force` の上書きは backup を取らない(`.scaffold-bak/` 等は提供しない)。ユーザが git 等でバージョン管理している前提。

## 5. 実装配置

### 5.1 新規ファイル

- `src/scaffold.lisp` — `defpackage cl-harness/src/scaffold`。テンプレート文字列・detection ロジック・ファイル書き出し・スカラー API すべてここに集約。
- `tests/scaffold-test.lisp` — rove deftests(§7 参照)。

### 5.2 既存ファイル変更

- `src/main.lisp` — facade に `#:scaffold` / `#:scaffold-result` / `#:scaffold-result-status` / `#:scaffold-result-paths-written` / `#:scaffold-result-conflicts` / `#:scaffold-error` / `#:scaffold-bad-system-name` / `#:scaffold-partial-state` を `:import-from` + `:export` で追加。
- `src/cli-main.lisp` — clingon の command tree に `scaffold` サブコマンドを追加。`fix` / `bench` / `develop` と並列の位置。`--project-root` / `--system` / `--test-system` / `--test-file` / `--force` を `clingon:make-option` で宣言。
- `cl-harness.asd` — `:cl-harness/src/scaffold` を `:cl-harness/src/main` の `:depends-on` チェーンに追加(facade が re-export するため)。`cl-harness/tests` 側に `:cl-harness/tests/scaffold-test` を追加。

### 5.3 依存関係

- `uiop` のみで足りる(パス操作、ディレクトリ作成、文字列操作)。
- `alexandria` は既に他所で使用されているが、scaffold.lisp 単体では不要。
- LLM / MCP / network いずれにも非依存。

## 6. エラーハンドリング

### 6.1 Condition 階層

```lisp
(define-condition scaffold-error (error)
  ((message :initarg :message :reader scaffold-error-message))
  (:report (lambda (c s) (write-string (scaffold-error-message c) s))))

(define-condition scaffold-bad-system-name (scaffold-error)
  ((name :initarg :name :reader scaffold-bad-system-name-name)))

(define-condition scaffold-partial-state (scaffold-error)
  ((existing :initarg :existing :reader scaffold-partial-state-existing)
   (missing  :initarg :missing  :reader scaffold-partial-state-missing)))
```

### 6.2 CLI 層の exit code マッピング

```
scaffold-bad-system-name  → exit 2
scaffold-partial-state    → exit 1 (--force でなければ)
any other condition       → exit 2 (unknown failure)
:already-present          → exit 0
:written                  → exit 0
```

### 6.3 プログラム呼び出し

scaffold 関数は condition を raise する(swallow しない)。呼び出し側が `handler-case` で分岐できる。

## 7. テスト戦略

`tests/scaffold-test.lisp` に下記 deftest を最低限揃える。すべて LLM mock 不要の純関数テスト。`uiop:with-temporary-directory` を使って tmpdir に対して走らせる。

1. **fresh-scaffold-writes-four-files** — 空 tmpdir に scaffold、4 ファイルが書かれ各内容が期待 substring(`(:class :package-inferred-system)` / `(:nicknames #:<system>)` / `(:use #:cl #:rove)` / `*.fasl`)を含む。
2. **idempotent-on-already-scaffolded** — 同じ tmpdir に 2 回 scaffold、2 回目は `:already-present` でファイル mtime が変わらない。
3. **refuses-partial-state-asd-only** — `.asd` だけ手で書いた tmpdir に scaffold、`scaffold-partial-state` が raise され `existing` slot に asd path、`missing` slot に src/tests path。
4. **force-overrides-partial-state** — 同状況で `:force t`、`:written` 返却、3 ファイル全部新規内容。
5. **bad-system-name-rejected** — `Demo` / `foo_bar` / 空文字 / `0name` を渡すと `scaffold-bad-system-name` raise。
6. **creates-project-root-if-missing** — 存在しない深いパスを渡すと `ensure-directories-exist` で作られる。
7. **generated-asd-loads-via-asdf** — scaffold 後に `(asdf:load-asd <path>)` が成功する。
8. **generated-defpackage-forms-parse** — `eclector:read-from-string` 等で 3 つの `.lisp` ファイルの defpackage フォームが構文エラーなく読める。
9. **test-system-override-honored** — `:test-system "demo/integration"` を渡すと .asd の test-system 名と tests/main-test.lisp の defpackage 名が両方とも反映される。
10. **system-name-as-nickname** — `:system "foo-bar"` → src の defpackage に `(:nicknames #:foo-bar)` が出る。

合計 ~10 deftests。LLM プロバイダ stub は不要。

## 9. Out of scope

本 spec の範囲外:

- **`:export` の自動生成** — planner 側で対応(`2026-05-22-planner-symbols-to-export-design.md`)。
- **`develop` 内での自動 scaffold チェーン** — 採用しない方針(§1.1 参照)。
- **複数 src ファイル / モジュール構造のテンプレート** — 単一 `src/main.lisp` のみ。複雑なレイアウトは v 次以降の検討。
- **`:license` / `:author` / `:description` の対話入力** — 全てプレースホルダ("TODO" 等)で出し、ユーザが後で編集。
- **`scaffold` の dry-run モード** — テンプレートが固定なのでメリット薄。YAGNI。
- **既存 .asd を解析して欠けたファイルだけ補う** — 部分存在 refuse の方針と矛盾するため意図的に未対応。

## 10. リスクと緩和策

| リスク | 緩和策 |
|---|---|
| `--force` でユーザの既存 src が消える | README + scaffold コマンドの help 文で「backup 取らない」を明示。CLI 層で `--force` 指定時に stderr に警告 1 行を出す。 |
| package-inferred-system 以外を使いたいユーザ | 現 CLAUDE.md が package-inferred-system を必須としているので scope 外。将来の design で classic `:defsystem` テンプレート選択を追加検討。 |
| system 名 regex が CL 規約より厳しい / 緩い | `^[a-z][a-z0-9-]*$` は実用範囲を網羅(全 develop-benchmarks fixture が合致)。境界ケースが出たら拡張。 |
| 生成 .asd の `:perform` が古い rove API に依存する | 100-greet で実証済の `uiop:symbol-call :rove :run` 形式を使う。rove の major bump があれば追従。 |

## 11. 実装順序の見立て

writing-plans skill で詳細化するが、おおよその区切り:

1. `src/scaffold.lisp` + condition 定義 + テンプレート関数 + scaffold 本体
2. `tests/scaffold-test.lisp` で 10 deftest
3. `cl-harness.asd` 更新 + `src/main.lisp` facade re-export
4. `src/cli-main.lisp` に clingon サブコマンド追加
5. README.md / PRD 追記
6. mallet clean、`asdf:compile-system :force t` で warning ゼロ確認

mallet / コミット粒度の指針は cl-harness 既存の慣習に従う。
