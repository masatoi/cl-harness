# v0.1.0-mvp post-tag audit / v0.2 roadmap

**Date**: 2026-05-06
**Trigger**: ground-truth audit of PRD coverage and harness practical
utility after the `v0.1.0-mvp` tag (`a410edc`).
**Conclusion**: capability MVP (PRD §17) is structurally closed but the
v0.1 tag **does not deliver a usable coding-agent harness for anything
beyond benchmark demos**. v0.2 needs ~3-5 days of focused work to close
the gaps below.

### §11 execution flow

§11.1 (fix command's 14-step flow): all 14 steps present, but step 14
("save transcript / metrics / patch diff") only saves transcript.

§11.2 (runtime-native OBSERVE → ORIENT → PROBE → PATCH → VERIFY loop):

is free to skip probing in any mode. 003-missing-import × runtime-native
hits max-turns specifically because the prompt's probe encouragement
distracts the model from the load-failure pointer; this is observable in
the Phase 5.1 transcripts.

## 2. Design issues (zero-base review)

Beyond the PRD checklist, things I'd structurally redesign if rebuilding:

### A. ASDF source-registry global rewrite for sandbox isolation

**Current**: `(asdf:initialize-source-registry '(:source-registry (:tree X) :ignore-inherited-configuration))` injected into the agent worker via `repl-eval`.

**Problem**:
- Mutates global ASDF state in the worker. Subsequent non-bench uses see a wedged registry.
- Incompatible with `pool-kill-worker` mid-run (registry override evaporates), which is why bench ships with `:clean-verify-p nil`.
- Multi-task runs depend on the same wedge being re-applied per task.

**Fix**: get cl-mcp to add an `:asd-path` argument to its `load-system` tool and stop touching the source-registry. Until that lands, do `pool-kill-worker reset=t` followed by `(asdf:load-asd <sandbox>/x.asd)` per task and accept that clean-verify won't have the registration after the kill — at minimum, document the limitation in the bench docstring (currently silent).

### B. Plain functions where PRD asked for generics

`summarize-tool-result` is the worst offender. New tools require editing the central dispatch table in `agent.lisp` and the schema-hints function in two places. Should be `(defmethod summarize-tool-result ((tool-name (eql :run-tests)) result) ...)` so users can extend.

### C. No shell CLI

`(cl-harness:fix ...)` only callable from a SBCL REPL. No clingon / unix-opts dependency. CI integration (`cl-harness fix --project ./demo --system foo`) is impossible. PRD §8.1 says CLI; we shipped REPL functions and called it CLI.

### D. No LLM-side action format guarantee

JSON action is a free-form chat content with markdown-fence stripping. We don't use OpenAI's `tools` API or `response_format: {"type":"json_object"}`. The Phase 1 `gpt-oss-20b` 400 ("Tool choice is none, but model called a tool") is a direct symptom: had we used the tools API, that error class wouldn't exist and we'd get structured output for free.

Trade-off: the tools API ties us harder to OpenAI semantics. But we could detect provider capability via `tools/list` and downgrade to chat-content JSON only when needed.

### E. agent-state / bench-result counters too narrow

`agent-state` slots: `turn`, `status`, `final-verify`, `final-action`, `token-total`, `patch-count`, `patch-attempts`. **Missing**: `tool-call-count`, `file-read-count`, `repl-eval-count`, `mcp-error-count`, `action-parse-error-count`, `started-at`, `last-event-at`. These are exactly the fields REQ-AGENT-003 limits and REQ-LOG-002 metrics need.

`run-limits` has the limit fields but `agent-state` lacks the matching counters — the asymmetry is why limits aren't enforced.

### F. No context compaction strategy

Every turn appends `(make-chat-message "user" tool-result-summary)` to the running message list. `complete-chat` gets the whole list every time. Phase 5.2's variance demo (003 trial 3: 49k tokens / 20 turns) is this, plus reasoning overhead. With a 128k-token model we don't crash, but cost grows linearly and informational density drops.

Minimum: a hard turn-window (e.g. keep system + initial-user + last 6 turns) or a summarization step every N turns.

### G. No parse-error retry budget

`parse-action` failures feed the error back as a user message and consume a turn. If the LLM emits malformed JSON 20 times in a row, that's 20/20 turns wasted. Should give-up after 3 consecutive parse errors.

### H. cl-mcp tool argument schemas hardcoded

`tool-schema-hints` in `agent.lisp` hand-wrote schemas for ~9 tools. cl-mcp returns full schemas via `tools/list` (we call it during initialize, then ignore the body). The hand-written schemas drift if cl-mcp changes a tool. Should fetch + format from `tools/list` on first turn.

### I. `+source-mutating-tools+` hardcoded

`("lisp-patch-form" "lisp-edit-form" "fs-write-file")` baked in. cl-mcp could grow new mutating tools (e.g. a hypothetical `lisp-rewrite-package`); we'd silently fail to auto-reverify. The cl-mcp tool description ought to carry that flag.

## 3. Practical utility (zero-base, per use case)

| Use case | Current readiness | Blockers |
| -------- | -----------------:| -------- |
| Hobby use: small CL project, single failing test, run from REPL | **6 / 10** | non-determinism (turns 5-20 same task), no shell CLI, no per-workflow shaping |
| CI integration: PR-triggered automated fix attempt | **2 / 10** | no shell CLI, no exit-code semantics, no dry-run, wall-clock not enforced, no structured failure codes |
| Research benchmarking | **5 / 10** | runner works but `:clean-verify-p nil` in bench means numbers are "fix worked in dirty image", not "fix worked from clean reload"; metrics too thin for ablations; multi-trial mostly unrun |
| Production auto-fix service | **1 / 10** | no rate limiting, no transient-error retry, no sandboxing across users, no auth model for cl-mcp, no cost cap |
| Self-dogfooding (run cl-harness on its own bug) | **untested** | same cl-mcp HTTP server shared by both; sandbox interaction with the dev's editor REPL untested |

The 10/10 single-trial benchmark headline is a **capability proof**.
It is not a stability claim and should not be cited as one.

## 4. v0.2 roadmap

Priority order. Effort estimates assume continued single-developer
pace.

### Tier 1 — credibility (do these before quoting any number)

1. **Enforce all `run-limits` fields** (~1 day)
   Add counters to `agent-state` (tool-call-count / patch-attempts /
   patch-applies / file-read-count / repl-eval-count / started-at).
   Check at turn boundary, return `:limit-exhausted` with the field name.
   Update `format-final-report` and `bench-result`.

2. **Track and emit REQ-LOG-002 metrics** (~half day)
   Once the counters exist (item 1), serialise them into the JSONL
   `:run-end` event and into `bench-result-*`.

3. **Patch diff logging (REQ-PATCH-003)** (~half day)
   When a source-mutating tool returns isError=false, capture the
   tool's own diff text or recompute via `diff` against the sandbox.
   Add a `:patch` event to JSONL.

### Tier 2 — usability (real users)

4. **Shell CLI** (~half-1 day)
   Add `clingon` (or `unix-opts`) dep. Generate a `cl-harness`
   binary via `(asdf:make :cl-harness)` with `:build-operation
   program-op`. Argv: `fix --project ... --system ...`.

5. **`--dry-run` (REQ-CLI-003)** (~half day)
   Run the agent loop but stub `call-tool` with a logger; only the
   LLM is exercised. Useful for prompt iteration without burning MCP.

6. **Parse-error retry budget** (~1 hour)
   Track consecutive `action-parse-error` count on agent-state, abort
   after 3 with `:status :give-up :reason :parse-loop`.

### Tier 3 — methodology (so that benchmarks mean something)

7. **Run the 10 task × 3 condition × 3 trial sweep** (~1 hour code, ~1 hour wall clock)
   Use the existing `run-benchmark-task-trials`. Fold variance into
   the markdown report.

8. **Resolve clean-verify in bench mode** (~1 day, blocked on cl-mcp)
   Either ship a tiny patch to cl-mcp that lets `load-system` accept
   `:asd-path`, or refactor the harness to use a per-task pool reset
   plus explicit `load-asd`. Without one of these, benchmark numbers
   carry the dirty-image asterisk.

9. **Probe-friendly tasks** (~half day)
   The current 10 tasks are mostly local-source-edit bugs. Add 2-3
   that genuinely benefit from REPL probing (e.g. a CLOS dispatch
   ambiguity that needs `(method-combination ...)` introspection,
   a numeric overflow that needs `(repl-eval (foo 2^32))` to
   reproduce). Until then, runtime-native vs generic-mcp comparisons
   are unfair to runtime-native.

### Tier 4 — design hardening (so the harness keeps working as cl-mcp evolves)

10. **Dynamic tool schema** (~half day)
    Use `tools/list` output instead of hardcoded `tool-schema-hints`.

11. **`summarize-tool-result` as generic** (~1 hour)
    Convert the central cond to `defmethod` keyed on the tool name as
    a keyword. `define-summarizer` macro for ergonomics.

12. **Context compaction** (~1-2 days)
    Keep system + initial + last K turns; summarize older turns into a
    single user message. Critical once we run anything beyond toy
    fixtures.

### Out of v0.2 scope (revisit at v0.3)

- Workflow subpackages REQ-WF-002..005 (package / macro / CLOS / ASDF
  specific shaping). Phase 5 evidence suggests one good workflow is
  better than four bad ones. Defer.
- Multi-model comparison. Mechanically supported now; just expensive.
- LLM-side action format via OpenAI tools API. Big architectural change;
  risk of regressing currently-working models.

### Estimated total v0.2 effort

Tier 1: ~2 days. Tier 2: ~2 days. Tier 3: ~2 days (incl. a cl-mcp
upstream PR). Tier 4: ~3 days.

**Tier 1 alone takes the harness from "demo" to "I can quote the numbers".**
That is the smallest cut that justifies a v0.2.0-rc1 tag.

## 5. Calibration on the v0.1.0-mvp claim

The annotated tag message says PRD §17 is closed. Re-reading PRD §17:

> 10 個以上の benchmark cases がある
> 3 condition 比較ができる

Both literally true (10 tasks shipped; runner supports 3 conditions).
But the spirit of those bullets — "the harness can be used to compare
agent strategies on Common Lisp tasks" — requires items 1 / 2 / 3 / 7
above. **The v0.1.0-mvp tag is a capability checkpoint, not a research
checkpoint.** That distinction was not clear in the tag annotation or
the README; clarifying it is the cheapest single deliverable.
