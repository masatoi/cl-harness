# cl-harness v0.4.0

**Theme:** From "TDD-Green helper" to a Common Lisp runtime-native
development harness — explicitly modelling the bottom-up,
REPL-driven, ledger-keeping development culture as part of the
agent loop.

**Date:** 2026-05-06

**Predecessor:** v0.3.0 (planner+orchestrator stack on top of
run-agent; greenfield fixtures 100-greet / 101-double).

**Reference:** `docs/notes/2026-05-06-v0.4-development-harness.md`
maps every requirement in `docs/development-requirement.md` to the
phase that delivered it.

---

## Highlights

### Phase 0 (shipped early as v0.3.1) — executor prompt 补强

The `:runtime-native` system prompt now teaches the executor:

- `lisp-edit-form` `form_name` is the form **body name**, not the
  form type — explicit examples for `defpackage`, `in-package`,
  `defun`, `defmethod` (with specializers).
- The greenfield insertion recipe: `defpackage` → `in-package` →
  first `defun` via `insert_after`.
- Fallback: after two consecutive `form_name` failures on the same
  target, switch to `fs-write-file` with a full-file replacement.

This was a prerequisite — without it Qwen3-class models could not
converge on greenfield fixtures, blocking every subsequent phase's
live verification.

### Phase 1 — Enriched plan-step schema

`plan-step` gains six optional slots:

- `purpose` — one-sentence why the step exists.
- `acceptance-criteria` — concrete checks beyond the rove test.
- `investigation-targets` — list of `{kind, name, intent}` records.
  `kind` ∈ `:package` `:symbol` `:function` `:generic-function`
  `:method` `:class` `:macro` `:system` `:test-system`.
- `risks` — likely failure modes (capped at three).
- `needs-exploration` — `:none` / `:lightweight` / `:deep`.
- `adopted-abstractions` / `rejected-abstractions` — written by
  the orchestrator after the explore phase.

The planner system prompt is updated to emit these. Older planner
responses without the new fields keep parsing — every new field
is opt-in.

### Phase 2 — Project inventory snapshot

Before each `plan-development` call, `gather-project-inventory`
walks the project root and produces a 5KB read-only snapshot
covering:

- Top-level `.asd` system / test-system structure.
- `src/*.lisp` heads (defpackage + first few forms collapsed).
- `tests/*.lisp` heads (defpackage + first few forms collapsed).

Sections are emitted in importance order with a per-file budget
of 480 chars. The inventory is prepended to the planner's user
prompt so the LLM builds on existing vocabulary instead of
inventing parallel structures.

### Phase 3 — Explore phase

When `plan-step.needs-exploration` is `:lightweight` or `:deep`,
the orchestrator runs an `run-explore-agent` first under a new
**`:explore` policy** that grants read-only tools only:
`repl-eval`, `code-find`, `code-describe`, `code-find-references`,
`inspect-object`, `lisp-read-file`. All write tools (`lisp-edit-form`,
`lisp-patch-form`, `fs-write-file`) are denied; attempts produce a
clear error in the JSONL.

The explore agent's terminal action is `finish` with a one-paragraph
memo. That memo is captured on `develop-step-result-explore-result`
and prepended to the implement step's issue as a
`## Prior exploration (read-only)` block, so the executor inherits
the discovery as plain text in its initial user prompt.

### Phase 4 — Abstraction ledger + markdown report

Explore memos are parsed for `ADOPTED:` / `REJECTED:` / `DEFERRED:`
markers. The parser tolerates:

- Em-dash (`—`), en-dash (`–`), colon (`:`), and surrounding hyphen
  (`-`) as separators between name and rationale.
- Leading bullets (`-`, `*`, `+`) and indentation.
- `(none)` sentinels (skipped silently).

Decisions are aggregated into `develop-result-abstraction-ledger`
and emitted as `:abstraction-decision` events on the develop-level
JSONL. The new `cl-harness:format-develop-report-markdown` renders
a structured markdown report with sections for user request,
outcome, adopted / rejected / deferred abstractions, exploration
notes per step, per-step status, and logs.

### Phase 5 — Integration check

After every step in a develop run reaches `:passed`, a static
cross-package consistency check parses every `.lisp` file under
`project-root` (with `*read-eval*` bound to `nil` for safety),
builds a defpackage graph, and flags two issue kinds:

- **`:unknown-package`** — `:import-from` names a package whose
  first segment matches an in-project package but is not defined
  anywhere in the graph.
- **`:unexported-symbol`** — `:import-from` imports a symbol the
  source package's defpackage does not list under `:export`.

Out-of-project deps (`alexandria`, `rove`, ...) are skipped via a
same-prefix heuristic — only packages whose first segment matches
an in-project package are evaluated. Issues are stored on
`develop-result-integration-issues`, emitted as an
`:integration-check` JSONL event, and rendered as their own
section in the markdown report. A clean check produces a single
"all consistent" line so the section is always visible.

### Phase 6 — Mode selector

`cl-harness:develop` and the shell `--mode` flag now take one of:

- **`:top-down`** — implement-first. The orchestrator coerces
  every plan-step's `needs-exploration` to `:none` before
  execution; the explore sub-agent never runs.
- **`:bottom-up`** — explore-first. `:none` and `nil`
  needs-exploration are promoted to `:lightweight`; explicit
  `:deep` is preserved.
- **`:mixed`** (default) — let the planner decide per step.

The planner also sees a `Mode: ...` line in its user prompt so a
well-behaved LLM produces a plan whose `needs-exploration` values
already match. The orchestrator's mechanical normalisation is the
second line of defence — misaligned plans get corrected.

`+supported-development-modes+` enumerates the keywords;
unsupported values raise `planner-error`.

### Phase 7 — Greenfield benchmark suite

`develop-benchmarks/` grows from 2 to 7 fixtures, covering a wider
shape range:

| Task | Shape |
|---|---|
| 100-greet | pure function |
| 101-double | pure function |
| 102-counter-class | defclass + 2 methods |
| 103-fizz-buzz | control flow + strings |
| 104-cache-simple | defclass + 2 generic functions |
| 105-validate-email | small parser |
| 106-format-currency | pure function + edge cases |

`src/develop-bench.lisp` adds a small loader API, also re-exported
from the `cl-harness` facade:

- `develop-task` class with id / goal / system / test-system /
  test-file / fixture-path.
- `load-develop-task` (path) — parse a single `develop-task.json`.
- `discover-develop-tasks` (suite-dir) — walk a directory, return
  parsed tasks sorted by id.
- `prepare-develop-task-sandbox` (task &key into) — copy
  `fixture/` to a fresh tmpdir (epoch + monotonic counter so
  concurrent calls do not collide); caller handles deletion.

A turnkey multi-model `develop-bench` runner is on the v0.4.1
wish-list; numbers will land in
`docs/benchmarks/results-2026-05-06-develop.md` (skeleton already
in place).

## Compatibility

All v0.3.0 entry points keep their kwarg / flag shapes. Every
v0.4.0 addition is opt-in:

- `cl-harness:fix` and `cl-harness:bench` are unchanged.
- `cl-harness:develop` accepts new kwargs (`:mode`,
  `:project-inventory`, `:gather-inventory-p`,
  `:inventory-byte-budget`); defaults preserve v0.3 behaviour.
- The shell `cl-harness develop` adds `--mode`; defaults to
  `mixed`.
- Existing `develop-task.json` files keep working — the loader
  only requires `id` / `goal` / `system` / `test-system` /
  `test-file`. `fixture-dir` defaults to `fixture`.

The v0.3 invariants carry over unchanged: clean-room ASDF
scoping, clean-verify on `:passed`, error_text in JSONL on tool
failures, the seven run-limits slots, stdio MCP transport by
default.

## Testing

- **184 unit tests** pass on a clean worker (v0.3.0 had 125;
  delta = +59 across Phase 1–7).
- mallet clean across all `src/` and `tests/` files.
- `(asdf:compile-system :cl-harness :force t)` produces zero
  warnings beyond the harmless ASDF `perform` redefinition that
  every package-inferred-system reload triggers.

## Known gaps

- `develop-bench` runner (multi-model orchestration) is still a
  shell-script affair — wrap `cl-harness:develop` per
  (task × model × mode) in a loop. Native support is a v0.4.1
  task.
- Multi-model baseline numbers for the 7-task greenfield suite
  are pending — `docs/benchmarks/results-2026-05-06-develop.md`
  is a skeleton waiting for runs.
- Auto-trigger for `compact-history` (Tier 4 C-3, shipped in
  v0.3.0 as a data-structure transform) inside the agent loop is
  still future work.

## Migration notes

No source migration required. To opt into v0.4 features:

```lisp
(cl-harness:develop
 :goal "..."
 :project-root "..."
 :system "..." :test-system "..."
 :test-file "..."
 :mode :bottom-up                ; explore-first; default :mixed
 :gather-inventory-p t           ; default; pass NIL to skip
 :inventory-byte-budget 5000     ; default
 ...)
```

The structured markdown report is now the recommended format for
post-run review:

```lisp
(let ((result (cl-harness:develop ...)))
  (princ (cl-harness:format-develop-report-markdown
          result :goal "..." :log-path log-path)))
```
