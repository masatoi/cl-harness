# cl-harness v0.5.0

**Theme:** From "harness with state" to "harness with structured
context" — every observation the agent loop gathers is now a typed
record on the central `develop-state`, and every LLM-facing prompt
is built from a phase-appropriate, compressed view of that state
rather than from raw transcript history.

**Date:** 2026-05-08

**Predecessor:** v0.4.0 (runtime-native development harness;
explore phase, abstraction ledger, integration check, mode
selector, 7-task greenfield benchmark suite).

**Reference:** `docs/context-management.md` is the requirements
doc; the §14 implementation-status table maps every section
(§3.1-3.9 / §4 / §5 / §6 / §8 / §9 / §10) to the phase that
delivered it.

---

## Overview

v0.5.0 ships the **context-management refactor**: 10 phases
(A → J) plus 4 follow-ups, covering every section of the
requirements document. The unifying theme is that the agent loop
no longer treats raw conversation history as authoritative —
every observation is captured as a typed record on `develop-state`,
and per-phase context views render exactly the slice each LLM
call needs.

The public CLI surface (`fix`, `bench`, `develop`) is unchanged.
All additions are opt-in via the existing `:develop-state` kwarg
on `cl-harness:develop` (now actively threaded into the planner),
or via the new structured-report formatter. v0.4 callers that
don't touch `:develop-state` see byte-identical behaviour from
their entry point.

## Highlights

### Phase A — Central `develop-state` class

`cl-harness/src/state` introduces a single mutable container that
threads through one `develop` invocation:

- Construction-time slots: `goal`, `project-root`, `system`,
  `test-system`, `condition`, `run-limits`, `project-inventory`,
  `mode`.
- Live slots: `current-plan`, `current-step-index`,
  `step-results`, `replan-count`, `last-failure-test-name`,
  `status`, `limit-hit`, `integration-issues`.
- Phases B–J extend this class with additional slots for each new
  ledger; the list grew to 7 by Phase J.

The orchestrator threads the `develop-state` through every helper
that previously took ad-hoc kwargs, and clears
`current-step-index` on step exit so recorders see "active step or
nothing".

### Phase B — `source-fact` / `patch-record` / `failure-ledger`

Three new ledger modules with matching slots on `develop-state`:

- **`source-fact`** (`src/source-fact.lisp`) — every successful
  `lisp-read-file` / `fs-read-file` / `clgrep-search` records the
  file path, optional form-type/form-name, the cl-mcp tool used,
  and `mtime-at-read` so a later staleness check can detect
  in-flight edits.
- **`patch-record`** (`src/patch-record.lisp`) — every successful
  source-mutating tool call records the path, form-type/form-name
  (when known), the tool used, the unified diff summary, the turn,
  and a verify-status state-machine slot.
- **`failure-ledger`** (`src/failure-ledger.lisp`) — active vs
  resolved partition over `failure-record` instances. The
  orchestrator's post-step hook moves prior-active failures whose
  test-name disappears from the latest verify-result into resolved,
  attributing each to the most recent patch on the same source
  file.

All three ledgers are populated automatically by the agent loop
and the orchestrator. No external code changes are required to
opt in.

### Phase C — `make-context-view` (3-phase formatters)

`cl-harness/src/context-view` introduces a `context-view` snapshot
class and a `context-view->string` generic with per-phase methods
for `:planning`, `:exploration`, and `:implementation`.

- **`:planning`** — goal, project inventory, prior-plan + failure-
  context (replan), now also a `## [STALE] Project summary` block
  (Phase I) and a `## Runtime vocabulary observed so far` block
  (Phase G) when those slots are populated.
- **`:exploration`** — current step's issue, investigation
  targets, source facts read in this step (with `[STALE]`
  annotation, Phase F), runtime vocabulary probed in this step
  (with `[STALE]`, Phase G), and findings observed in this step
  (with `[PROMOTED]` annotation when shipped, Phase H).
- **`:implementation`** — current step's issue, patches applied,
  active failures, completed subtask summaries (Phase J),
  recently resolved failures (regression watch, Phase J), and
  unpromoted findings (REPL-confirmed but not yet shipped,
  Phase H).

The planner / explore / agent prompt-builders all consult the view
when a `develop-state` is in scope; the legacy ad-hoc string
assembly is preserved for standalone callers / test stubs / bench
harness's planner-fn callback.

### Phase D — Tool result compression + `compact-history`

Seven cl-mcp tools' outputs (`fs-read-file`, `lisp-read-file`,
`clgrep-search`, `code-find`, `code-describe`,
`code-find-references`, `inspect-object`) are truncated at 1500
characters with a `(N more bytes truncated)` footer.
`summarize-run-tests` adds a `(N more failures truncated)` footer
when more than 5 failures are present. `compact-history` runs in
the agent loop's `step-turn` immediately before `complete-chat`,
gated by `run-limits.max-context-tokens` (default 50000).

### Phase E — Structured markdown reporting

`format-develop-state-report` (`src/report.lisp`) emits a
markdown report from a `develop-state`: Goal / Plan / Completed
steps / Patches applied / Active failures / Resolved failures /
Integration issues / Source facts. Empty sections elide. The
existing `format-develop-report` is preserved for v0.4 callers;
the new `format-develop-report-structured` wrapper is opt-in.

`source-fact-stale-p` predicate (the `[STALE]` rendering's
foundation) lands here and is consumed by Phase F at render time.

### Phase F — `[STALE]` annotation on source-facts

`:exploration` formatter renders stale source-facts (whose
on-disk mtime exceeds the recorded `mtime-at-read`) with a
`[STALE]` prefix at render time. `make-context-view` does not
inspect staleness; the predicate fires inside the formatter.
Annotate-not-filter: stale facts still render so the LLM stays
aware of all observations.

### Phase G — `runtime-vocabulary` ledger

A peer to source-fact for runtime introspection observations.
`runtime-vocab-fact` with `kind` ∈ `{:package :symbol :function
:generic-function :method :class :macro :condition :asdf-system}`,
plus name / package / source-file / summary / via-tool /
probed-at / related-step-index. The agent loop pushes facts on
successful `code-find` / `code-describe` / `code-find-references`
results.

The `:exploration` view annotates stale facts with `[STALE]`
(Phase F pattern extended); the `:planning` view shows a
warm-start summary alongside the existing project-inventory text.

`%vocab-kind-from-string` uses `find-symbol` (NOT `intern`) so
unknown kinds from external cl-mcp results never pollute the
keyword package — satisfies the project-style runtime-INTERN ban.

### Phase H — `repl-finding` ledger ("REPL success ≠ implemented")

Structured `(hypothesis probe finding decision)` 4-tuple records
captured from `{"type":"finding"}` actions emitted by the explore
sub-agent. Each finding carries a `promoted-to-source-p` flag
(idempotent `repl-finding-mark-promoted` setter) plus optional
`linked-patch` / `linked-source-fact` references.

Promotion linkage is automatic: the orchestrator's post-step path
walks not-yet-promoted findings and flips the flag when the
hypothesis text appears (case-fold substring) in any patch's
diff-summary.

The `:exploration` view lists ALL findings with `[PROMOTED]`
annotation on shipped ones; the `:implementation` view lists ONLY
unpromoted findings under "Findings to implement (REPL-confirmed,
not yet shipped)" — enforcing the §3.6 "REPL success ≠
implemented" rule at the view layer.

### Phase I — `project-summary` with dirty-on-asd/defpackage

A structured wrap of `gather-project-inventory` (UIOP-only
reader) with `asd-files` / `source-files` / `test-files` /
`text` / `gathered-at` / `dirty-p` slots. The agent loop's
post-patch hook (`%maybe-mark-summary-dirty`) flips `dirty-p`
when patch path matches `*.asd` (case-insensitive) OR when
patch form-type is `"defpackage"`.

The `:planning` view renders `## [STALE] Project summary` (the
`[STALE]` prefix added at render time when dirty) followed by
the file lists, BEFORE the existing `## Project inventory` text
block — augments, not replaces.

This is the FIRST single-value (not list-of-records) context-
management slot on `develop-state` (`:accessor` direct rather
than `%`-prefix-with-public-reader pattern).

### Phase J — Subtask summaries + regression watch

`subtask-summary` records (DERIVED, not stored on develop-state)
captured by `summarise-step-result` from existing step-results /
patch-records / abstraction-decisions. The `:implementation`
view renders prior `:passed` steps as `## Completed subtask
summaries` bullets, plus `## Recently resolved failures
(regression watch)` with the most-recent N (default 3, configurable
via `+resolved-failures-context-limit+`) resolved failures.

Both blocks emit ONLY when their list is non-empty: empty
develop-state produces byte-identical view output.

### Follow-up 1 — Step-index threading (cross-phase fix)

Phase H's final code review surfaced a cross-phase gap: all three
agent-loop recorders (source-fact / runtime-vocab / patch-record)
had been hard-coding `:related-step-index nil`, so per-step view
filters silently dropped production-recorded data (only
hand-built test fixtures with explicit step-index were visible).
The orchestrator already maintains `current-step-index`
(Phase A); the recorders just needed to read it.

`%record-runtime-vocab-from-tool-call` was extracted as a private
helper to enable a clean unit test that doesn't require the
broader tool-result parser.

### Follow-up 2 — Phase C wiring activated

The orchestrator's two `funcall planner-fn` call sites now pass
`:develop-state state`. Production planner prompts are built via
the `:planning` context-view formatter; the legacy ad-hoc string
assembly is preserved for callers that don't pass
`:develop-state`. Section ordering shifts slightly (mode-nudge
moves to AFTER the context-view block) but content is identical.

### Follow-up 3 — `%vocab-facts-from-tool-result` shape fix

A Phase G bug: `(gethash "results" result)` returns NIL when the
key is absent (code-describe shape), and `(listp NIL)` is T, so
code-describe results entered the empty-loop branch and returned
no facts. Fix: dispatch on `multiple-value-bind` to distinguish
"key absent" vs "key present (any value)". Code-describe and
code-find-references results now record runtime-vocab-facts in
production.

(The Phase J Task 2 merge commit attributed this to "yason
parses JSON arrays as vectors". Verification confirmed yason
returns CL lists for JSON arrays under defaults — the actual bug
was the absent-key vs empty-list ambiguity.)

### Follow-up 4 — `develop-step-result` extraction

Phase J Task 2 broke a load-time circular dependency
(`orchestrator → explore → context-view → subtask-summary →
orchestrator`) via `find-symbol` + `slot-value` runtime
resolution. This follow-up extracts `defclass develop-step-result`
into a dedicated `src/step-result.lisp` module (zero outbound
deps); `subtask-summary.lisp` returns to compile-time imports.
Slot-name typos are now compile-time errors. Public API
unchanged: `:import-from` shares symbol identity, so callers
writing `cl-harness/src/orchestrator:develop-step-result-X`
continue to work.

## Compatibility

All v0.4.0 entry points (`cl-harness:fix`, `cl-harness:bench`,
`cl-harness:develop`, the shell binary) keep their kwarg / flag
shapes. v0.5.0 additions are opt-in:

- `cl-harness:develop` now actively threads `:develop-state` to
  the planner (Follow-up 2). The planner's user-prompt section
  ordering shifts slightly but content is identical.
- `format-develop-report-structured` is the new opt-in
  ledger-aware report formatter; `format-develop-report` /
  `format-develop-report-markdown` remain unchanged.
- `develop-state` and its ledger slots / readers are exported
  but not re-exported from `cl-harness/src/main` — callers that
  want to introspect should `:import-from` the per-module
  packages directly.

The v0.4 invariants carry over unchanged: clean-room ASDF
scoping, clean-verify on `:passed`, JSONL transcript shape,
seven `run-limits` slots, stdio MCP transport by default,
explore phase + abstraction ledger + integration check + mode
selector all continue to work as before, just feeding a
richer context view.

## Migration notes

No source migration required. v0.4 callers see identical
external behaviour. To opt into v0.5 visibility:

```lisp
(let ((result (cl-harness:develop ...)))
  ;; New structured report (ledger-aware):
  (princ (cl-harness:format-develop-report-structured result))

  ;; Or per-ledger introspection:
  (let ((state (cl-harness::develop-result-develop-state result)))
    (cl-harness/src/state:develop-state-source-facts state)
    (cl-harness/src/state:develop-state-patch-records state)
    (cl-harness/src/state:develop-state-failure-ledger state)
    (cl-harness/src/state:develop-state-runtime-vocabulary state)
    (cl-harness/src/state:develop-state-repl-findings state)
    (cl-harness/src/state:develop-state-project-summary state)))
```

The `develop-state` is also exposed on the develop-result for
external readers (`cl-harness:develop-result-develop-state`).
