# cl-harness v0.5.1

**Theme:** Bug-fix + production-side wiring polish surfaced from
the v0.5.0 live verification with Qwen/Qwen3.6-35B-A3B. No new
ledgers, no new context-view phases — just making the v0.5.0
machinery actually work in production against reasoning models
and exercising the Phase G/H/I paths that were wired but unused.

**Date:** 2026-05-08

**Predecessor:** v0.5.0 (context-management refactor — 10 phases
A → J + 4 follow-ups, 357 unit tests).

---

## What changed

### Fix: dexador read-timeout 10s → 600s

dexador's default `*default-read-timeout*` is **10 seconds**
(verified via repl). That's fine for sub-second chat models
(Groq llama-3.3) but broke v0.5.0's first-call planner against
**reasoning models** like Qwen3.6 where a single LLM call takes
30-60s. The develop loop crashed on the first `complete-chat`
with `"I/O timeout while doing input on #<SB-SYS:FD-STREAM ...>"`
before any JSONL event could be written.

Added `+default-llm-read-timeout-seconds+` (600 = 10 min,
generous enough for slow first-prompt warmups without masking a
genuinely-stuck endpoint). `default-llm-transport` passes it as
`:read-timeout` to `dexador:post`. Exposed as a `defparameter` so
callers can adjust without touching the transport function.

After this fix, the same Qwen3.6 endpoint completes 100-greet
end-to-end in ~50s with `--reasoning-effort low`.

Also added `.claude/scheduled_tasks.lock` to `.gitignore`
(per-machine runtime artifact written by Claude Code's wakeup
scheduler).

### Phase G/H/I prompt-side enrichment

The v0.5.0 verification surfaced three "machinery exists but
production code path doesn't exercise it" gaps:

- **runtime-vocabulary stayed empty** under `:runtime-native` —
  Phase G's recorder triggered on `code-find` / `code-describe` /
  `code-find-references` results, but the system prompt didn't
  tell the LLM to use those tools.
- **repl-findings stayed empty** — the `{"type":"finding"}` action
  shape (parser, explore loop persistence, orchestrator
  promotion) was implemented but the explore prompt only
  documented `tool_call` and `finish`. The LLM had no way to know
  `finding` existed.
- **project-summary stayed NIL** — `gather-project-summary` was
  implemented but never called from the develop loop.

This release closes all three with **prompt-only changes** for G/H
and a **best-effort wiring** for I:

- `src/agent.lisp:system-prompt` `:runtime-native` mode gains a
  paragraph recommending `code-find` / `code-describe` /
  `code-find-references` as the first-choice probe for vocabulary
  questions, with explicit "the harness records each successful
  query as a runtime-vocab fact" framing. `:generic-mcp` and
  `:file-only` branches are byte-identical to v0.5.0.
- `src/explore.lisp:explore-system-prompt` action-schema list
  gains a `finding` action variant between `tool_call` and
  `finish`. Workflow gains a paragraph between rules 2 and 3
  explaining WHEN to emit a finding ("after a probe confirms a
  hypothesis") and the orchestrator's auto-promote semantics.
- `src/orchestrator.lisp:develop` calls `gather-project-summary`
  immediately after `make-develop-state`, wrapped in
  `handler-case (... (error () nil))` so a malformed project
  tree degrades to NIL rather than killing the loop.
  `develop-state-set-project-summary` populates the slot.

Tests: 357 → **361** (+4 deftests asserting keyword presence in
prompts and slot non-NIL after develop).

The implement run-agent's prompt is **deliberately untouched** —
the `finding` action is explore-only by design.

## Live verification (Qwen/Qwen3.6-35B-A3B)

Three scenarios on a local SGLang endpoint:

| Run | mode + condition | fixture | status | source-facts | patch-records | runtime-vocab | repl-findings | project-summary |
|---|---|---|---|---|---|---|---|---|
| A | mixed + generic-mcp | 100-greet | :passed | 1 | 1 | 0 | 0 | **PRESENT** |
| B | bottom-up + runtime-native | 100-greet | :passed (2 step, 2 adopted) | 2 | 1 | 0 | 0 | **PRESENT** |
| C | bottom-up + runtime-native + max-tokens=8192 | 102-counter-class | :limit-exhausted (3-step plan, step 0 hit turn budget) | 4 | 1 | 0 | 0 | **PRESENT** |

**Phase I auto-gather is deterministic** — populated on every
run. **Phase G/H prompt changes are present** but Qwen3.6 with
`reasoning_effort=low` continues to favour file-based tools
(`fs-list-directory` / `fs-read-file` / `lisp-read-file`) over
runtime introspection, and continues to terminate exploration
with `ADOPTED:` markers in the final memo rather than emitting
discrete `finding` actions during the loop. The prompts are in
place; behavioural change is gated on stronger prompting,
higher reasoning effort, or different model — orthogonal to
this release.

## Compatibility

All v0.5.0 entry points (`cl-harness:fix`, `cl-harness:bench`,
`cl-harness:develop`, the shell binary) keep their kwarg / flag
shapes. v0.5.1 is a non-breaking patch:

- The dexador timeout bump is transparent to callers — fast
  models (sub-second responses) see no behavioural change; slow
  reasoning models that previously crashed now complete.
- The Phase G/H prompts are additive — existing workflow lines
  unchanged. LLM behaviour change is opt-in via the LLM noticing
  the new guidance.
- Phase I's auto-gather populates a previously-NIL slot; callers
  that didn't read `develop-state-project-summary` see no
  change. Callers that read it now see a populated record.

The v0.4 / v0.5 invariants carry over unchanged: clean-room
ASDF scoping, clean-verify on `:passed`, JSONL transcript shape,
seven `run-limits` slots, stdio MCP transport by default,
explore phase + abstraction ledger + integration check + mode
selector all continue to work as before.

## Known gaps (carried from v0.5.0 + new)

- Qwen3.6 with `reasoning_effort=low` doesn't yet exercise the
  Phase G `code-*` probes or Phase H `finding` action despite
  the new prompt language. Stronger prompting / higher reasoning
  effort / different model evaluation remains a research task.
- `develop-bench` runner (multi-model orchestration) — same
  status as v0.5.0.
- The 4 develop-bench-test failures under shell `rove` (fixture
  path resolution) — same status, not specific to
  context-management.

## Migration notes

No source migration required. Existing v0.5.0 callers see:

- Fixed dexador timeout — slow LLM calls no longer crash on the
  first request.
- `develop-state-project-summary` is non-NIL after every
  `cl-harness:develop` run (was always NIL in v0.5.0).
- The `:planning` view rendered through `context-view->string`
  now includes a `## Project summary` block with the structured
  asd / src / tests file lists (was never visible in v0.5.0
  because the slot was NIL).

To explicitly disable the auto-gather (e.g., for benchmark runs
with synthetic project trees), call
`(cl-harness/src/state:develop-state-set-project-summary state nil)`
after develop returns; or wrap your `cl-harness:develop` call to
suppress the slot population if needed in your environment.
