# cl-harness v0.5.2

**Theme:** Production resilience for the LLM transport layer —
**typed failure-mode classification** end-to-end + a **minimal
automatic retry policy** for the most-clearly-transient failures.

**Date:** 2026-05-08

**Predecessor:** v0.5.1 (dexador timeout fix + Phase G/H/I prompt
enrichment, 361 unit tests).

---

## What changed

v0.5.2 lands two related but independent phases on top of v0.5.1.
Both came out of the same observation: production runs against a
real reasoning-model endpoint surface several distinct failure
modes (server 5xx, rate-limited 429, socket timeout, malformed
JSON, empty `content`), and the harness was conflating them under a
single "model raised" path. v0.5.2 separates them at the source
and adds a minimum-viable recovery for the genuinely-transient
ones.

### Transport failure-mode coverage

Replaces the ad-hoc string-match error path in `complete-chat`
with a typed `model-error` classification. Each transport / parse
failure now carries a `:kind` keyword that propagates from
`complete-chat` → orchestrator → `develop-result` so callers can
distinguish what actually went wrong without parsing error
messages.

- **Typed `model-error` `:kind`** — 8 keyword values cover the
  observed failure surface:
  - `:auth-failed` (HTTP 401 / 403)
  - `:http-server-error` (HTTP 5xx)
  - `:http-client-error` (other 4xx)
  - `:rate-limited` (HTTP 429)
  - `:transport-timeout` (`usocket:timeout-error`)
  - `:transport-unavailable` (`connection-refused-error`,
    generic `socket-error`)
  - `:malformed-response` (non-JSON or schema-violating body)
  - `:empty-content` (HTTP 200 with `content=null` or empty
    string — surfaces as `:give-up :empty-content` upstream
    rather than a hard error, since some reasoning models emit
    this when token budgets are too tight)
- **`:reason` slot end-to-end** — `agent-state`, `develop-state`,
  and `develop-result` all gained a `:reason` slot. The
  orchestrator catches `model-error` and stamps the kind on
  `develop-result-reason` so external callers (`fix`, `bench`,
  `develop` shell wrapper) can branch on the keyword without
  inspecting the condition object.
- **`tools/chaos-probe.lisp`** — manual real-LLM runner that
  drives three deliberately-broken scenarios (P1 empty-content
  via `max-tokens=1`; P3 transport-unavailable via unreachable
  port; P4 auth-failed via bad API key) and asserts the expected
  `:status` / `:reason` pair. Verifies the classification logic
  end-to-end against real endpoints, not just stubs.
- **Report formatters** render `:reason` in the structured
  markdown report and the JSONL transcript.
- **C3 length truncation** — when `finish_reason="length"`,
  `chat-parse-response` now forwards the partial `content` rather
  than dropping it, so the agent loop can still proceed with
  what the model managed to emit before the cap.

Stub-injectable `:transport` kwarg on `make-openai-provider` makes
the entire failure surface CI-testable: the new
`tests/transport-test.lisp` (29 deftests covering classification +
end-to-end propagation) runs deterministically without needing
a live LLM endpoint. Live verification via `chaos-probe.lisp`
remains a manual step against a configured endpoint.

Two reviewer follow-ups landed shortly after the merge as
non-blocking cleanups:

1. `step-turn-with-empty-content` agent-loop deftest — closes a
   coverage gap where the empty-content fast path was tested at
   `complete-chat` but not at the higher `step-turn` boundary.
2. `tools/chaos-probe.lisp` `ql:quickload` → `asdf:load-system` —
   removes a Quicklisp dependency that wasn't actually needed
   for the probe (the system is already on the local ASDF
   path).

Tests: 361 → **390** (+29 transport-test.lisp deftests +
follow-up).

### LLM retry policy

Adds a **minimal** automatic retry to `complete-chat` so genuinely
transient failures recover without bothering the caller. Surface
is deliberately tiny: 1 retry, no backoff, only on the 3
most-clearly-transient `:kind` values.

- **`+retriable-reasons+`** defparameter (internal) —
  `'(:http-server-error :rate-limited :transport-timeout)`.
  `:malformed-response` and `:transport-unavailable` are
  deliberately **excluded** until production data shows the
  false-positive cost of retrying them is acceptable. The
  reasoning is documented inline at the defparameter.
- **`retry-p` slot** on `openai-compatible-provider` (default
  **T**, reader `provider-retry-p`). NIL disables retry
  entirely.
- **`:retry-p` kwarg** on `make-openai-provider` and threaded
  through `cl-harness:develop` / `fix` / `bench` for caller
  control.
- **2-attempt retry loop** wraps the existing `complete-chat`
  body: initial attempt + 1 retry on the same retriable kind.
  No backoff (per the design's α decision — "smallest surface;
  covers the cheap transient case; can be expanded if production
  shows need"). The `url` / `headers` / `body` let-bindings sit
  outside the loop so retries reuse the encoded request.
- **`tools/chaos-probe.lisp` opts out** with `:retry-p nil` so
  its deliberately-broken scenarios don't double their wait
  time on retry.

Tests: 391 → **397** (+6 retry-policy deftests covering success
on each retriable kind, exhaustion, non-retriable propagation,
and the `:retry-p nil` opt-out).

The retry default is **on**: production callers transparently
gain protection against transient blips. Existing callers that
relied on raise-on-first-failure semantics (none in the codebase
today) can opt out per-provider. Per-call override was
considered and dropped as YAGNI.

## Compatibility

All v0.5 entry points (`cl-harness:fix`, `cl-harness:bench`,
`cl-harness:develop`, the shell binary) keep their kwarg / flag
shapes. v0.5.2 is a non-breaking enhancement:

- The typed `model-error :kind` is additive — existing callers
  that catch `model-error` continue to work; they just gain the
  ability to read `model-error-type` if they want to branch.
  Callers that read `develop-result-reason` see new keyword
  values rather than NIL on transport failure paths.
- The retry-by-default behavior change is benign for the
  current codebase: every caller path goes through
  `cl-harness:develop` / `fix` / `bench`, which thread
  `:retry-p`; tests that need single-shot semantics opt out
  via `%make-stub-provider`'s `:retry-p` parameter (default T,
  matching production); chaos-probe opts out scenario-locally.
- C3 length truncation now forwards content; previous v0.5.x
  behavior was to drop the partial response. Callers that
  inspect `chat-response-content` may now see non-empty strings
  on `finish_reason="length"` paths that previously came back
  empty. The `:reason` field still flags the truncation.

The v0.4 / v0.5 invariants carry over unchanged: clean-room
ASDF scoping, clean-verify on `:passed`, JSONL transcript shape,
seven `run-limits` slots, stdio MCP transport by default,
explore phase + abstraction ledger + integration check + mode
selector all continue to work as before.

## Known gaps (carried + new)

- `:malformed-response` and `:transport-unavailable` are
  excluded from the retry set pending production data on
  false-positive cost. Likely revisited in v0.5.3 or v0.6.
- Retry has no backoff. Fixed-delay or exponential backoff is
  the natural next phase if production shows real-world endpoint
  rate-limit shapes that need it. The `Retry-After` header is
  also not honored yet.
- Per-call retry override (vs the per-provider `:retry-p`) is
  YAGNI for now.
- JSONL events for retry attempts are not emitted — observability
  improvement deferred until production shows it's needed.
- Qwen3.6 prompt-following gaps from v0.5.1 remain unchanged
  (model behavior, not harness logic).
- The 4 develop-bench-test failures under shell `rove` (fixture
  path resolution) — same status as v0.5.0 / v0.5.1, not
  specific to this release.

## Migration notes

No source migration required. Existing v0.5.1 callers see:

- `complete-chat` raises `model-error` with a typed `:kind`
  instead of a generic error on transport failure paths.
  Callers that already caught `model-error` keep working;
  callers that inspected error messages with string-match
  should switch to `(model-error-type c)`.
- `develop-result-reason` populated with new keyword values on
  failure paths that previously returned `:status :error`
  with NIL `:reason`. Existing branches keying on
  `develop-result-status` continue to work; new keyword reasons
  are additive.
- The retry policy is on by default. To restore v0.5.1
  raise-on-first-failure semantics for a specific call path,
  pass `:retry-p nil` to the relevant CLI entry point or
  construct the provider with `(make-openai-provider :retry-p
  nil ...)` directly.
