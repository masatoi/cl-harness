# cl-harness v0.6.0

**Theme:** `develop`-loop resilience — eliminate silent fatal aborts
in the planner/orchestrator path and build a 3-layer defense around
`%plan-with-review`.

**Date:** 2026-05-28

**Predecessor:** v0.5.2 (typed `model-error` + minimal retry policy,
397 unit tests).

---

## TL;DR

v0.6.0 wraps the develop loop with three layered failure handlers
(planner-fn / deterministic / LLM review), all funneled to the same
replan request shape. The cumulative effect on the
`develop`-benchmarks fixtures we measure regularly:

| Fixture | N | Pass rate (pre) | Pass rate (post) | silent ERROR (pre) | silent ERROR (post) |
|---|---:|---:|---:|---:|---:|
| 100-greet | 10 | 100% | 100% | 0 | 0 |
| 104-cache-simple | 10 | 40% | **70%** | 4/10 | **0/10** |
| 102-counter-class | 10 | ~33% (history) | **60%** | ~1/3 | **0/10** |

Silent `:ERROR` terminal state from the `develop` loop is eliminated
across all benched fixtures. Non-trivial fixtures gain pass rate;
the trivial fixture's baseline is preserved with tighter variance.

Public CLI surface (`fix` / `bench` / `develop` / `scaffold`) is
unchanged — all v0.6.0 behavior changes are improvements inside
existing entry points with no kwarg / flag breakage.

## What changed

### 3-layer defense in `%plan-with-review` (Findings 6+7)

`%plan-with-review` used to trust both that `planner-fn` returns a
well-shaped plan and that `execute-plan`'s own L1 validator never
encounters a bad `test_source`. When Qwen-style reasoning models
exhaust their token budget mid-response, both assumptions break in
production: the JSON gets truncated, a deftest form gets cut off at
an unbalanced paren. The bare `planner-error` then escaped the
develop loop's outer `HANDLER-CASE` (which only catches
`MODEL-ERROR`) and aborted the run.

`%plan-with-review` now wraps the loop in three layered handlers,
all converted to the same "ask the planner for a fresh plan" shape:

- **P0 — `planner-fn` itself** (Finding 7): a `PLANNER-ERROR`
  raised inside `funcall planner-fn` (yason JSON decode failure,
  schema violation, etc.) is caught and replanned with the
  planner-error message as failure-context.
- **L1 — deterministic structural gate** (Finding 6):
  `%validate-plan-test-sources` runs `validate-test-source` on
  every step's `test_source` after planner-fn returns. Failures
  are collected into a feedback string and converted into a
  replan request, instead of letting `execute-plan` raise.
- **L2 — LLM plan/tests review** (existing): unchanged; runs
  after L1 passes.

Budget exhaustion (`max-review-replans`) lands on
`:LIMIT-EXHAUSTED` — a visible terminal state — instead of
silent `:ERROR`.

### `test_change_request` hardening (DR-2026-05-27)

Implementation review on 2026-05-27 surfaced five gaps in the
`test_change_request` path that landed via the v0.5.x review-policy
work. All five closed:

| # | Issue | Severity | Fix |
|---|---|---|---|
| H1 | name-collision check L2-only | High | new `%extract-deftest-names-from-file` + `validate-test-source` returns test name as 2nd value + deterministic collision check |
| M1 | `(rove:deftest …)` substring/validator inconsistency | Medium | removed `(search "(deftest " source)`; trust validator's structural result; docstring documents qualified-form env dependency |
| M2 | L1 structural reject didn't reach agent feedback loop | Medium | `%maybe-handle-test-change-request` wraps approve branch in `HANDLER-CASE` → converts `planner-error` into `:REJECTED <feedback>` |
| L1 | `develop-state-test-revision-count` docstring stale | Low | updated to attempted/consumed budget semantic |

The earlier 5-finding design review (Findings 1–5) had already
landed via commits d6fa5a9, b3ef5b1, b1fee38 between v0.5.2 and the
implementation review. v0.6.0 ships the consolidated state of all
ten findings.

### Tolerant package read in `validate-test-source`

The d6fa5a9 fix that introduced structural Lisp reading in
`validate-test-source` was latent-broken for greenfield fixtures:
planner output legitimately references the new system's package via
qualified symbols (e.g. `(deftest fizz-buzz::sample (ok (= 1
(fizz-buzz:fizz-buzz 1))))`), and that package doesn't yet exist in
the worker image. `READ` blew up with `Package FIZZ-BUZZ does not
exist.` and every greenfield fixture deterministically aborted.

`%pre-create-referenced-packages` scans the source before `READ`,
finds `NAME:` / `NAME::` prefixes, and ensures each `NAME` exists as
a bare `:USE nil` package with the trailing symbol interned (and
exported for single-colon `NAME:SYM`). Side effects are bounded to
the ephemeral sandbox worker — leaked empty packages don't outlive
the bench run.

Three smoke-test fixtures went from 0% pass (deterministic abort)
to 100% pass on re-test after this fix.

### Bench-driven improvement cycle

Between v0.5.2 and v0.6.0 the project ran ~10 paired / sweep bench
cycles against Qwen3.6-35B-A3B to identify and verify the highest-
leverage defensive improvements. The work that made it into v0.6.0:

- **`:max-consecutive-failed-patches`** (#45, default 3): catch
  patch oscillation within a step before the existing
  `:max-patches` budget; agent can't drag the loop indefinitely
  when the same patch keeps reverting.
- **`:max-stalled-verify-cycles`** (#56, default 3): catch
  verify-without-progress (same `failed_tests` shape across N
  consecutive verifies); agent loop ends with `:STUCK` instead
  of burning all turns.
- **Empty-content retry** (#40): `complete-chat` retries once on
  empty-content responses from reasoning-model token exhaustion;
  exhausted retries land on `:give-up` with `reason
  :empty-content` instead of throwing a less-readable
  `model-error`.
- **Stage-aware review prompts** (#54/#55): the review system
  prompt is now `:default` / `:tests` / `:test-change`-specific
  (strict for tests-touching kinds, soft for implementation),
  preventing legitimate plans from being rejected by an
  over-aggressive reviewer.
- **Planner: one new exported symbol per step** (#41/#50): the
  planner prompt now nudges decompositions toward one new
  undefined symbol per step, so each verify failure is
  interpretable as "this single symbol still missing" rather
  than "multiple symbols missing simultaneously".
- **`:predefined-plan` kwarg** (#46 core): bypass the planner LLM
  call entirely. Used by the paired-bench infrastructure to
  measure fix-on / fix-off effect of develop-loop changes
  without LLM variance contaminating the measurement.
- **#38 patching-guidance removal**: a v0.5.x experimental
  prompt segment was found net-negative by N=10 paired bench
  (`docs/benchmarks/results-2026-05-24-38-paired-n10-104b.md`)
  and removed in 1f29778.

### `max-context-tokens` default tightened

Lowered from 50000 to 4000 (a2191ee). Reduces token spend per LLM
call by ~26% on baseline `100-greet` traces without measurable pass-
rate impact.

### Test count

498 unit tests passing on a clean cl-mcp worker (v0.5.2: 397;
delta +101 across the DR-2026-05-27 fix series + Finding 6+7 +
the bench-driven defensive limits).

## Migration notes

No breaking changes. Every kwarg / flag accepted by v0.5.2 still
accepts the same values and means the same thing in v0.6.0.

The two notable behavior changes that callers might observe:

- `develop` runs that previously aborted with `:ERROR` on a
  planner-fn JSON decode failure or a malformed `test_source` now
  retry the planner (up to `max-review-replans`). Budget exhaustion
  surfaces as `:LIMIT-EXHAUSTED` with `limit-hit :max-review-replans`.
- `max-context-tokens` default is now 4000. Set it back to 50000 if
  you were depending on the prior larger window for very long traces.
