# v0.4 Development Harness Plan

要件: `docs/development-requirement.md`

cl-harness を「failing test を直す道具」から「Common Lisp の
REPL駆動・ボトムアップ開発文化を AI エージェントの実行ループに組み込んだ
development harness」に拡張するための、 段階的開発計画。

本 note は実装の commitment ではなく、 採用前のレビュー対象として
書く。 数値の見積もりや責務分担は確定ではなく、 工事を始める前の
合意点を作るためのもの。

---

## 1. 要件 → 現在地マップ

| 要件 | 現状 (v0.3.x) | 不足分 |
|---|---|---|
| 4.1 Planning | `plan-step` に issue / test_name / test_source / files_to_modify | **目的・受け入れ条件・調査ターゲット・リスク・探索要否** が無い |
| 4.2 Orchestration | execute-plan + develop (replan + stuck 検出) | **探索フェーズの dispatch / 統合チェック** が無い |
| 4.3 Runtime Inspection | cl-mcp の code-find / code-describe / inspect-object はあるが、 develop ループから **明示的に使われていない** | **プロジェクトの語彙を要約して planner に渡す** 仕組みが無い |
| 4.4 REPL Exploration | cl-mcp の repl-eval は executor で使えるが、 **「探索のための独立フェーズ」** として呼び出される構造が無い | 同上 |
| 4.5 Source Promotion | 手続きとして存在しない | **REPL 試作 → 採用 / 不採用判断 → source 反映** の明示的ステップが無い |
| 4.6 Abstraction Control | 何も記録しない | **採用 / 不採用の抽象** を ledger として残す仕組みが無い |
| 4.7 Implementation | run-agent で done | (基本的に充足。 ただし greenfield で `lisp-edit-form` 周りに弱さがあるので executor prompt 補強が要る) |
| 4.8 Test Generation | planner が test_source を 1 件 author | **integration test や既存挙動の non-regression test** までは扱えていない |
| 4.9 Integration | 各 step が独立、 統合フェーズなし | package export / ASDF deps の整合性チェックが無い |
| 4.10 Clean Verification | clean-verify-task で done | (充足) |
| 4.11 Reporting | develop-result + 二段 JSONL | **採用抽象・残課題・探索要約** が出ていない |
| 5.1 Top-down mode | 既存 `develop` がほぼこれ | (準) |
| 5.2 Bottom-up mode | 無い | 探索を先に走らせる経路が必要 |
| 5.3 Mixed mode | 部分的 (default が top-down のみ) | step ごとの explore-first / implement-first 切替が無い |

## 2. 設計上の前提と方針

### 2.1 既存 v0.3.0 のアーキテクチャを壊さない

`cl-harness:fix` は今のまま、 既存ユーザは無影響。 `develop` も後方互換を保ち、 拡張は新 kwarg として opt-in にする。

### 2.2 探索フェーズは「executor の中で発生する」 ものではなく、 「orchestrator が明示的に呼ぶ」 ものとする

理由: executor の中で「探索 vs 実装」 を判断する責任を持たせると、 1 turn ごとにモード切替の判断を LLM にさせることになり、 turn budget を食い、 且つ判断の一貫性が出ない。 orchestrator が「この step は探索が要る」 と明示し、 executor は「探索専用の policy + 別の system prompt」 で動く方が clean。

### 2.3 抽象化判断は LLM curate、 ledger は構造データ

採用 / 不採用は **LLM の判断** で記録するが、 **何を採用したか** と **何を見送ったか** はそれぞれ struct スロットに残す (RAG ではなく explicit log)。 後で人間が読んで「この決定は妥当か」 を判断できる。

### 2.4 model-capability gap の明示的扱い

要件 4.7 の Implementation を greenfield で完遂するには、 現状 Qwen3.6-35B では `lisp-edit-form` の使い方を間違える。 v0.4 のロードマップとは別に **executor の system prompt を補強する** 並行作業が必要。 このプランは「モデル能力が満ちている前提で、 ハーネス側の設計を整える」 もの。

## 3. フェーズ計画 (草案)

各フェーズは独立してテスト・review 可能な単位で切る。 v0.3.x → v0.4.0 までの粒度。

### Phase 0 (v0.3.1, prerequisite) — 1〜2 日

executor の system prompt に `lisp-edit-form` insert_after の form_name 例を追加。 `(defpackage ...)` / `(in-package ...)` への insert_after の書式を 1〜2 件サンプル化。 同時に「同じ form_name で 2 連続失敗 → fs-write-file 全置換に切り替えろ」 のヒントを system prompt に追加。

これは v0.4 開発に取り掛かる前に **greenfield fixture を実機で `:passed` まで通す** ことが先決。 通らないと以後のフェーズの数値が取れない。

### Phase 1 (v0.4 P1) — 2 日: Enriched plan-step schema

`plan-step` に次のスロットを追加 (オプション):

- `purpose` — 1 文
- `acceptance_criteria` — string list
- `investigation_targets` — list of `{kind, name, intent}`、 例 `{:kind :function :name "demo:greet" :intent "確認したい既存 API"}`
- `risks` — string list (max 3 想定)
- `needs_exploration` — keyword: `:none` / `:lightweight` / `:deep`
- `adopted_abstractions` / `rejected_abstractions` — orchestrator が後で書き戻すスロット

planner の system prompt を更新して、 これらを emit する。 後方互換: 旧フィールドのみの応答も受理 (新フィールドは NIL)。

### Phase 2 (v0.4 P2) — 3 日: Project Inventory snapshot を planner に渡す

plan-development を呼ぶ前に、 cl-mcp の以下を順に叩いてスナップショットを作る:

- `fs-get-project-info` でディレクトリ構造
- `code-find` で各パッケージの export 関数 / 主要 generic / class
- `lisp-read-file collapsed=true` で .asd の system / test-system 構造

これを「Project Inventory: ...」 という名前で planner の user-prompt に注入する。 planner は知っている語彙の中で **既存設計に乗る** 計画を立てる。

トークンサイズ管理: 5KB 上限を目安にトリム。 大きいプロジェクトは大きい summarization が要るが v0.4 はとりあえず固定 prefix.

### Phase 3 (v0.4 P3) — 3 日: Explore phase

orchestrator に新しい sub-phase を追加: `%execute-explore-step`。

`plan-step.needs_exploration` が `:lightweight` 以上なら、 implement の前に「探索 run-agent」 を 1 つ走らせる。 探索 run-agent は:

- policy `:explore` (新しい mode)、 ツール:
  - `repl-eval`、 `inspect-object`、 `code-find`、 `code-describe`、 `code-find-references`
  - `lisp-read-file` (read-only)
  - **書き込み系は禁止** (lisp-patch-form / fs-write-file 不許可)
- 結果として **「探索結果メモ」 を 1 段落** 返してくる
- このメモが implement step の system prompt に **prior knowledge** として注入される

explore の中で発見した **採用候補抽象** は plan-step.adopted_abstractions に 蓄積。 採用しなかったものも記録 (4.6)。

実装メモ: explore step も deftest を materialize するわけではない。 **観察のみ**。

### Phase 4 (v0.4 P4) — 2 日: Abstraction ledger + reporting

`develop-result` に `abstraction-ledger` スロットを追加。 各 plan-step の adopted / rejected を集計。 `format-develop-report-markdown` を新規追加し:

```
## ユーザー要求
...

## 採用した抽象
- defclass logger / log-level slot — 既存 stream-event 設計に沿う
- defun log-info — 通常関数で十分

## 見送った抽象
- defmacro with-log-context — まだ使い道が 1 箇所、 既存の dynamic let で代替可能
- defgeneric log-event — class はまだ 1 個、 必要になったら昇格

## 探索した内容
- repl-eval で stream-event の export を確認、 既存 export 表に従って logger を配置
...

## 残課題
- log rotation は未実装、 別 issue 化想定
```

を出す。 同時に develop-level JSONL に `abstraction-decision` イベント種別を追加。

### Phase 5 (v0.4 P5) — 2 日: Integration check

全 step が `:passed` した後、 1 つの追加 step として「integration check」 を走らせる:

- 全 source ファイルの defpackage `:export` 一覧を集計
- 全 test ファイルの `:import-from` を集計
- 不整合 (export されていないシンボルを import しようとしている、 未使用 export、 など) を検出
- 全 test を最終的にもう一度 clean-verify

検出された不整合があれば最終 1 round の replan を許可。

### Phase 6 (v0.4 P6) — 2 日: Mode selector

cli:develop に `:mode` kwarg を追加: `:top-down` / `:bottom-up` / `:mixed` (default)。

- `:top-down`: needs_exploration を全部 `:none` に矯正、 直接 implement
- `:bottom-up`: planner に「最初は実装より探索を優先」 と instructive な system prompt 補強、 plan-step.needs_exploration default `:lightweight`
- `:mixed`: planner が step ごとに自由に decide

shell CLI にも `--mode` を追加。

### Phase 7 (v0.4 P7) — 2 日: Greenfield benchmark suite + 数値固定

`develop-benchmarks/` を 5–8 件に拡張 (現状 2 件):

- 100-greet (既存)
- 101-double (既存)
- 102-counter-class (defclass + 2 methods)
- 103-fizz-buzz (制御フローと文字列)
- 104-cache-simple (defclass + 2 generic functions)
- 105-validate-email (small parser)
- 106-format-currency (純粋関数 + edge cases)

それぞれ複数モデル (Qwen3, Claude Sonnet 4.6, GPT-4o-mini など API 取れたもの) で develop を回し、 v0.3 ベンチに並ぶ baseline を取る。 `docs/benchmarks/results-XXXX-develop.md` に固定。

## 4. ロードマップ (フェーズの並び)

```
v0.3.1: Phase 0 (executor prompt 補強で greenfield を Qwen3 で通す)
v0.4.0:
  Phase 1 (plan schema)
  Phase 2 (inventory)
  Phase 3 (explore)
  Phase 4 (abstraction ledger + report)
  Phase 5 (integration)
v0.4.1:
  Phase 6 (mode selector)
  Phase 7 (benchmark suite + 数値固定)
```

各 v0.4.x マイナーが ~1 週間想定。 全部で 4〜6 週間規模 (実時間ではなく、 1 セッション数時間ペースの累積)。

| 受け入れ条件 (§7) | 達成フェーズ | 検証方法 |

| 既存 fix / bench を壊さない | 全フェーズ | 既存 125 unit tests |

## 6. オープン論点 (採否の議論が要る)

1. **inventory snapshot のサイズ**: 5KB 固定か、 LLM の context window に応じてダイナミックか? 大規模プロジェクトでは structured summary を別 LLM call で作る必要あり。 v0.4 では固定、 v0.5 で動的化。
2. **explore step の出力構造**: 自由文 1 段落 vs 構造化 (発見したシンボル list、 試した s-expression、 結論)。 後者の方が確実だが prompt token が増える。
3. **abstraction-ledger の決定主体**: planner 案 / explore 案 / 別の reviewer LLM。 v0.4 は planner と explore の両方が emit、 重複排除は orchestrator。
4. **integration check の責任範囲**: package export 一致だけにするか、 macro expansion / 循環 dep 検出まで踏み込むか。 v0.4 は前者のみ。
5. **bottom-up mode の "ボトムアップらしさ"**: planner に "探索を優先" と言わせるだけで足りるか、 それとも別の planner system prompt を用意するか。 prompt-engineering の実験タスク。
6. **既存 benchmark との関係**: bug-fix benchmarks (`benchmarks/`) は develop 用にも使えるか? 既存テストがあるので explore は要らない、 純 implement 系。 v0.4 で develop が bug-fix にも適用可能か検証する。

## 7. 非目標 (要件 §6 + 本プラン)

- multi-agent 並列、 GUI、 Emacs 連携、 GitHub 自動化、 vector DB、 sandbox security
- macro / DSL の自動生成 (4.6 で「特に慎重に」 とある通り、 v0.4 は **生成しない方向**)
- 大規模リファクタリング自動化
- Slack / メール通知

## 8. 短期に始めるなら

依存関係を考えると、 **Phase 0 (v0.3.1) を最優先**。 これが終わらないと Phase 3 の explore も live verify できない。

Phase 0 の中身は小さい:
- executor の system prompt に「lisp-edit-form の form_name は form の本体 (defpackage の名前、 in-package の引数の package designator)、 form_type ではない」 と明示する 3〜4 行の補強
- 同じ form_name で連続失敗時の fallback 案 (fs-write-file 全置換) を明示
- 1 件の greenfield fixture (100-greet) を Qwen3 で通すまで iteration

これが収束したら Phase 1 から順次。

## 9. 振り返り基準

各フェーズ完了時の "shippable" 判定:

- 既存 unit tests が全 PASS
- 新フェーズ固有のテストが少なくとも 5 件追加
- mallet clean
- ライブで対応するシナリオが少なくとも 1 回 :passed する
- 設計ノート or README に動作仕様が反映されている

これらが揃わないフェーズは v0.4.x の release candidate に含めない。
